# 第一章 产品概述

## 1.1 产品背景

随着家族结构趋于小型化与分散化，传统的纸质族谱或静态电子家谱已无法满足现代家庭记录动态成长轨迹与维系情感连接的需求。用户需要一个轻量、易用、可持续的工具，以数字化方式记录家族成员的鲜活故事与集体活动，实现家族记忆的"活态传承"。

## 1.2 产品定位

一款聚焦于动态记录与情感连接的家族记忆轻应用。核心在于帮助家庭便捷、持续地记录个人成长与家族共同经历，构建有温度的数字化家族记忆库。

**产品slogan：** 让记录家族故事，像发朋友圈一样简单。

## 1.3 核心价值主张

| 维度 | 价值主张 |
|------|----------|
| 对个人 | 轻松记录人生轨迹，便捷回顾成长历程 |
| 对家庭 | 通过共同记录活动（如扫墓、聚会），增强代际互动与情感纽带 |
| 对家族 | 系统性沉淀家族故事与文化，形成可传承的数字资产 |

## 1.4 目标用户

| 用户群 | 特征描述 |
|--------|----------|
| 主要用户 | 25-50岁的家族中坚成员，有记录和维系家庭关系的意愿，熟悉智能手机操作 |
| 次要用户 | 愿意接触新事物的长辈，及被邀请参与记录的青年一代 |

## 1.5 术语表

| 术语 | 定义 |
|------|------|
| 家族码 | 系统为每个家族生成的唯一6位数字/字母组合，作为快速加入家族的凭证 |
| 家族树 | 可视化展示家族成员关系的树状结构图 |
| 成员卡片 | 记录家族成员基本信息的数字化档案卡，含必填字段（姓名、性别、出生日期、肖像照）和选填字段 |
| 事件卡片 | 记录在个人时间轴上的单条动态内容单元 |
| 家族活动 | 由家族成员共同参与并记录的集体事件，如扫墓、聚会、旅行等 |
| 守护人 | 经创建者指定、拥有最高数据管理权限的1-2位成员，可在线索人账号失效时代替管理 |
| 外姓成员 | 因婚姻关系（配偶）加入家族树、姓氏与本家族不同的成员节点 |
| 关系类型 | 成员间关系类别，含：血缘（blood）、婚姻（spouse）、结拜（sworn）、认亲（adoptive）四类 |
| 辈分 | 根据家族树中的代数位置自动计算的宗族排行 |
| 称谓 | 根据关系类型和辈分自动生成的亲属称呼 |
| 可见性级别 | 信息的可见范围控制：公开（所有成员）、亲属可见（直系及旁系）、仅自己可见 |
| 事件标签 | 动态记录的分类标签：工作、学业、家庭、健康、成就、旅行、其他 |
| 简易模式 | 面向长辈用户的简化界面，字体更大、功能更少、操作路径更短 |
| 家族动态墙 | 聚合展示所有家族成员近期个人动态与家族活动的信息流页面 |
| 那年今日 | 系统定期推送的、过往同日期家族活动或个人动态回顾内容 |

## 1.6 产品边界与排除项

### 1.6.1 产品边界（本版本涵盖）

- 仅支持微信小程序平台（iOS + Android）
- 以单个家族为核心数据单元，支持用户同时加入多个家族
- 支持血缘、婚姻、结拜、认亲四类关系
- 记录形式：文字（≤300字）、图片（≤3张）、短音频（≤60秒）
- 使用微信云开发作为后端服务

### 1.6.2 明确排除项（本版本不涵盖）

| 排除项 | 排除原因 | 规划版本 |
|--------|----------|----------|
| 跨平台支持（APP/H5） | MVP聚焦小程序体验 | V2.0评估 |
| 族谱打印/实体印刷服务 | 需要供应链对接 | V3.0+ |
| AI自动生成文章/故事整理 | 需要额外AI能力接入 | V3.0+ |
| 家庭级社交互动（跨家族评论/点赞） | 隐私与产品定位考量 | V3.0+ |
| 多人实时协作编辑同一条动态 | 复杂度高，MVP优先级低 | V2.0 |
| 活动地点3D地图可视化 | 地图API成本与开发量 | V2.0 |
| 语音输入转文字 | 需要语音识别能力 | V2.0 |
| 完整的旅行活动管理（行程规划） | 超出家族记录核心场景 | V1.5 |

**关键转化指标：**

| 阶段 | 目标 | 验收标准 |
|------|------|----------|
| 注册 → 创建家族 | ≥ 70% | 登录流程顺畅，无阻断 |
| 创建家族 → 添加3位成员 | ≥ 50% | 引导清晰，操作简单 |
| 添加成员 → 首次记录 | ≥ 40% | 3步内完成记录发布 |

## 2.2 新手引导流程

**新手任务（首次启动）：**

1. **欢迎页** → 产品介绍短视频（15秒）/ 跳过按钮
2. **微信授权** → 获取昵称头像（如拒绝，允许手动填写姓名）
3. **创建/加入家族** → 二选一：创建新家族 / 输入家族码加入
4. **引导添加成员** → 建议添加至少3位家庭成员（父母/配偶/子女）
5. **首次记录引导** → 推荐使用模板发布第一条动态
6. **开启提醒授权** → 引导开启订阅消息（如拒绝，不阻断使用）
7. **新手任务奖励** → 完成全部任务，解锁「记录达人」称号徽章

**长辈/新手关怀：**
- 首次进入自动检测简易模式（可在设置中切换）
- 关键操作配有操作指引气泡（可关闭）
- 文字提示配合图标，降低理解门槛

# 第三章 功能详细设计

## 模块一：家族管理

### 1. 功能概述与价值

家族管理模块是整个应用的骨架，提供家族创建、成员管理、关系绑定和权限控制等核心能力。它决定了数据的组织方式，是所有其他功能的基础。

### 2. 用户故事（User Story）

| # | 角色 | 故事描述 | 验收条件 |
|---|------|----------|----------|
| US-F01 | 家族创建者 | 作为家族发起人，我希望一键用微信登录并创建家族，以便快速启动家族记忆的记录 | 微信授权后3步内完成家族创建并获得家族码 |
| US-F02 | 普通成员 | 作为家族成员，我希望通过家族码快速加入家族，以便与家人共享记忆 | 输入正确家族码并审核通过后可见家族全部内容 |
| US-F03 | 家族管理员 | 作为管理员，我希望在族谱上直观地看到所有成员关系，以便维护家族信息的准确性 | 家族树清晰展示所有成员，支持缩放和拖拽导航 |
| US-F04 | 普通成员 | 作为普通用户，我希望添加我的配偶信息并看到她在我家族树中的位置，以便完整记录家庭 | 配偶以"外姓成员"形式显示在族谱中，姓名前有标识 |
| US-F05 | 家族创建者 | 作为创建者，我希望为家族设置2位守护人，以便在我无法使用账号时有人接管 | 最多设置2位守护人，创建者可随时变更 |

### 3. 功能流程与交互细节

#### 3.1 创建与加入家族

**创建流程：**
1. 微信一键登录（静默获取openid，显示微信昵称头像）
2. 输入家族名称（必填，2-20字）
3. 选择家族姓氏（选填，用于辈分排序）
4. 系统生成6位家族码（数字+字母混合）
5. 生成邀请链接与分享海报
6. 引导添加首批成员（建议3位）

**加入流程：**
1. 输入家族码或扫描二维码
2. 查看家族基本信息（名称、成员数、创建时间）
3. 提交加入申请
4. 由管理员/创建者审核（可设置免审）
5. 审核通过后进入家族

**家族码规则：**
- 格式：6位，数字+大写字母（如：AB3C9D）
- 同码唯一：一个码对应一个家族
- 可重置：管理员可更换家族码（旧码保留24小时）

#### 3.2 可视化家族树

**交互规范：**

| 操作 | 行为描述 |
|------|----------|
| 双指捏合缩放 | 族谱整体缩放（最小0.5x，最大3x） |
| 单指拖拽 | 移动族谱画布位置 |
| 点击成员节点 | 弹出成员卡片浮层（姓名+照片+关系+快捷操作） |
| 长按成员节点 | 进入编辑模式（管理员可见编辑/删除选项） |
| 上滑/下滑 | 顺/逆时间顺序展开（按出生年排列） |
| 节点展开/折叠 | 点击"+"展开子女节点，点击"-"折叠 |

**族谱布局规则：**
- 默认以最年长一代为顶部，按「左父右母」原则水平排列
- 子女节点居中显示在父母节点下方
- 辈分越低，节点越靠下
- 配偶节点与本人节点水平相邻，姓名前标注「配偶」

#### 3.3 成员卡片

**必填字段：**

| 字段 | 类型 | 规范 |
|------|------|------|
| 姓名 | 文本 | 1-10字 |
| 性别 | 单选 | 男/女 |
| 出生日期 | 日期 | 需早于当天，不可晚于当天；已故成员可填"未知" |
| 肖像照 | 图片 | 比例1:1，建议≥200×200px，最多上传3张 |

**选填字段：**

| 字段 | 类型 | 规范 |
|------|------|------|
| 职业 | 文本 | 0-30字 |
| 联系方式 | 文本 | 手机号/邮箱，需设置可见性 |
| 常住地 | 文本 | 省市区的文本描述 |
| 生平简介 | 多行文本 | 0-500字，已故成员可填写较详细的生平 |
| 微信昵称 | 文本 | 已在平台注册成员的微信昵称，自动同步 |

#### 3.4 关系绑定

**关系类型与称谓规则：**

| 关系类型 | relation_type | 典型称谓 | 说明 |
|----------|---------------|----------|------|
| 血缘-父母 | blood/father, blood/mother | 父亲、母亲、爸爸、妈妈 | 自动计算辈分为上一代 |
| 血缘-子女 | blood/son, blood/daughter | 儿子、女儿 | 自动计算辈分为下一代 |
| 血缘-兄弟姐妹 | blood/sibling | 哥哥、弟弟、姐姐、妹妹 | 同代旁系 |
| 婚姻-配偶 | spouse | 丈夫、妻子、配偶 | 外姓成员，姓名前标注「配偶」 |
| 结拜 | sworn | 义兄、义弟、义姐、义妹、干兄弟 | 虚线连接，不影响辈分 |
| 认亲 | adoptive | 干爸、干妈、干儿子、干女儿 | 可挂接任意成员下 |

**辈分自动计算规则：**
1. 以家族创建者（或最早成员）为基准代（代数为0）
2. 每向上一代加1，向下一代减1
3. 配偶、结拜、认亲关系不参与本姓辈分计算
4. 称谓由系统根据辈分+性别自动生成，用户可手动修改显示称谓

#### 3.5 族谱外延扩展逻辑

**结婚（婚姻关系）：**
- 配偶自动以"外姓成员"形式入谱
- 姓名前显示「配偶」标签或特殊边框
- 不参与本姓辈分排序，但在族谱中保留关系链
- 若配偶也有独立家族树，可设置「双向关联」，在配偶族谱中同步显示
- 新增配偶时询问：「该成员是否已有独立家族？」（可选关联）

**结拜/义兄弟：**
- 关系类型标记为「sworn」
- 在族谱中以虚线框或特殊图标（🤝）区分
- 结拜关系与血缘关系在视觉上分层显示
- 不自动计算辈分，称谓由用户手动选择（义兄/义弟/义姐/义妹）
- 允许设置结拜年份作为关系备注

**认亲/干亲：**
- 关系类型标记为「adoptive」
- 可挂接在任意成员下作为「干子女/干父母」
- 称谓自动生成（干爸/干妈/干儿子/干女儿），用户可手动修改
- 不影响原生辈分计算
- 允许设置认亲时间作为关系备注

#### 3.6 权限管理

**角色分级：**

| 角色 | 权限说明 |
|------|----------|
| 创建者（超级管理员） | 拥有全部权限，可转让家族、解散家族（需确认）、设置管理员/守护人 |
| 管理员 | 管理成员（添加/移除/审核）、编辑所有成员信息、管理活动、修改部分设置 |
| 普通成员 | 编辑自己的成员卡片、发布动态、参与活动、浏览家族内容 |
| 游客（待审核） | 仅可见家族名称和基本信息，不可浏览内容，审核拒绝后清除 |

**信息可见性分级：**

| 级别 | visibility值 | 可查看范围 |
|------|-------------|-----------|
| 公开 | public | 所有成员可见（含已故成员） |
| 亲属可见 | family | 创建者、管理员、直系亲属（父母/子女/配偶）及旁系亲属可见 |
| 仅自己可见 | private | 仅本人可见（创建者和管理员在紧急情况下可申请查看） |

**权限验证规则：**
- 每次数据访问前验证用户角色与信息可见性
- 敏感操作（如删除成员、修改关系）需二次确认
- 操作日志记录，留存180天

### 4. 边界情况与错误处理

| 场景 | 处理方式 |
|------|----------|
| 家族码已失效/不存在 | 提示「家族码不存在或已失效，请核实后重新输入」 |
| 申请加入被拒绝 | 提示「您的加入申请已被拒绝，可能原因：xxx」，可重新申请 |
| 成员信息填写不完整 | 阻断发布，红色高亮缺失字段，提示「请完善必填信息」 |
| 关系绑定冲突（如已设置父母关系） | 提示「您与该成员的关系已存在」，提供修改入口 |
| 删除有动态记录的成员 | 二次确认「该成员有X条动态记录，删除后将不可恢复」，可选保留记录（匿名化） |
| 创建者退出家族 | 引导先转让创建者身份，否则不允许直接退出 |
| 守护人被移除成员身份 | 自动取消守护人资格，需重新指定 |
| 配偶双向关联的两个家族互相冲突 | 提示冲突节点，需管理员手动解决 |

| AC编号 | 验收条件 | 测试方法 |
|--------|----------|----------|
| AC-F01 | 微信授权登录成功后在3步内完成家族创建并获得家族码 | 创建新账号，模拟完整创建流程，记录步骤数 |

## 模块二：动态记录

### 1. 功能概述与价值

动态记录模块是产品的核心创新点，旨在降低记录门槛、鼓励持续记录。通过「记一下」快捷入口、模板引导和智能提醒，让记录家族故事变得像发朋友圈一样简单，同时比朋友圈更有结构、更有温度。

### 2. 用户故事（User Story）

| # | 角色 | 故事描述 | 验收条件 |
|---|------|----------|----------|
| US-D01 | 普通用户 | 作为记录者，我希望通过首页的「记一下」按钮3步内完成一条记录，以便随时随地捕捉生活瞬间 | 点击「记一下」→ 选择模板 → 填写内容 → 发布，全程≤3步 |
| US-D02 | 忙碌的家长 | 作为双职工家庭的家长，我希望通过语音快速记录孩子的成长趣事，以便在碎片化时间也能保持记录 | 支持60秒内短音频记录，识别成功后转存为音频事件卡片 |
| US-D03 | 长辈用户 | 作为不太会打字的长辈，我希望看到引导性问题来帮助我想起该记录什么，以便轻松完成记录 | 记录页展示引导问题，如「最近孩子有什么新变化？」 |
| US-D04 | 关注健康的用户 | 作为关注健康的用户，我希望记录我的健康数据并设置周期提醒，以便养成健康管理习惯 | 支持健康日记模板，可设置每日/每周提醒 |

### 3. 功能流程与交互细节

#### 3.1 个人时间轴

**页面结构：**
- 顶部：成员切换器（查看自己/其他成员时间轴）
- 中部：事件卡片流（按时间倒序排列）

```
├─────────────────────────────────────────┤

├─────────────────────────────────────────┤
│ 💬 3条评论    🤝 5次共鸣                 │
└─────────────────────────────────────────┘
```

#### 3.2 快捷记录（「记一下」）

**入口位置：** 底部Tab栏中央，悬浮大按钮，视觉突出

**核心操作路径（≤3步）：**

```
步骤1：选择模板（或直接开始自由记录）
步骤2：填写内容（文字+图片+音频）
步骤3：确认发布
```

**记录模板清单：**

| 模板名称 | 引导问题 | 预设标签 | 默认字段 |
|----------|----------|----------|----------|
| 工作新变动 | 「最近工作有什么新进展？」 | 工作 | 成就标签 |

| 自由记录 | 无引导，用户自行编辑 | 其他 | 无 |

#### 3.3 记录形式规范

| 形式 | 限制 | 压缩规则 |
|------|------|----------|
| 文字 | 最多300字 | 无需压缩 |
| 图片 | 最多3张，单张≤2MB | 自动压缩至80%质量，长边≤1920px |
| 短音频 | 最多1条，≤60秒 | 格式转换至AAC，码率64kbps |

**图片上传流程：**
1. 点击「添加图片」→ 选择来源（相册/拍照）
2. 选择图片后自动压缩，显示压缩进度条
3. 最多可添加3张，超出后按钮置灰
4. 支持拖拽调整顺序（第一张为封面图）

**音频录制流程：**
1. 点击「按住说话」开始录制，松手结束
2. 显示录制时长进度条，最长60秒
3. 录制中可点击「×」取消
4. 完成后可预览播放，支持重新录制

#### 3.4 智能提醒与引导

**周期性提醒：**

- 提醒时间：用户偏好时间（默认晚8点）
- 提醒内容：「[张三]，好久没更新了，今天有什么想分享给家人的吗？」

**场景化引导问题：**
- 在记录页固定展示引导问题
- 每30天轮换一次问题库
- 引导问题示例：
  - 「最近有什么让你开心的小事？」
  - 「孩子学会了什么新技能？」
  - 「家人有什么值得记录的变化？」
  - 「今天有什么小成就？」
  - 「家庭聚会上有什么难忘的瞬间？」

**「那年今日」推送：**
- 每天早9点推送（可关闭）
- 内容：「[张三]，1年前的今天，您记录了：[摘要]」
- 点击可跳转到历史记录详情页

### 4. 边界情况与错误处理

## 模块三：家族活动与仪式

### 1. 功能概述与价值

家族活动模块聚焦于记录家族集体记忆，特别是扫墓这一在中国文化中具有重要意义但现有工具很少专门支持的活动。通过标准化活动模板和专项功能，让集体活动的记录更加完整和有仪式感。

### 2. 用户故事（User Story）

| # | 角色 | 故事描述 | 验收条件 |
|---|------|----------|----------|
| US-A01 | 家族组织者 | 作为清明扫墓的组织者，我希望记录祭扫对象、祭品和现场氛围，以便完整保留祭祖记忆 | 创建扫墓活动，关联祭扫对象，记录祭品和现场描述，上传墓碑照片 |
| US-A02 | 分散的家人 | 作为在外地工作的家族成员，我希望通过活动相册看到家人的相聚瞬间，以便感受家族温暖 | 活动相册展示所有参与者的照片，支持滑动浏览 |
| US-A03 | 关心长辈的用户 | 作为关心长辈的用户，我希望能设置已故亲人的忌日提醒，以便在重要日子及时问候 | 设置忌日提醒（农历/公历），每年自动推送 |
| US-A04 | 组织聚会的用户 | 作为家庭聚会组织者，我希望**直接指定参与成员（无需对方确认）**，并由每位参与者**各自发布心得与照片、内容互不覆盖**，以便形成可追溯的共同回忆 | 创建活动并勾选参与者即入列；多位参与者分别提交心得与配图，后端独立存储、互不影响 |

### 3. 功能流程与交互细节

#### 3.1 活动创建流程（6步）

**步骤1：选择活动类型**

| 活动类型 | 图标 | 适用场景 |
|----------|------|----------|
| 扫墓 | 🕯️ | 清明、重阳、日常祭扫 |
| 家庭聚会 | 🎉 | 春节、中秋、生日宴等 |
| 节庆 | 🎊 | 传统节日庆祝 |
| 婚礼寿宴 | 🎊 | 婚礼、寿宴等 |
| 旅行 | ✈️ | 家族出游（V1.5完整功能） |
| 自定义 | 📝 | 其他活动 |

**步骤2：填写基础信息**

| 字段 | 必填 | 规范 |
|------|------|------|
| 活动标题 | 是 | 2-30字 |
| 活动时间 | 是 | 日期+时间选择器 |
| 活动地点 | 否 | 文本输入，支持地图选址（腾讯地图API） |
| 活动描述 | 否 | 0-200字 |

**步骤3：关联参与者**

- 从家族树成员列表中勾选
- 支持全选/反选
- 显示已选人数：「已选择 8 位参与者」
- **发起方（组织者）直接指定参与人员：被勾选成员即视为本次活动参与人员，无需成员端「同意/确认」流程。**
- ~~被邀请但未确认的成员以灰色显示~~（**已废止**：本产品不做「待受邀人确认」状态；若需展示「未实际到场」等，由后续版本以「到场/缺席」等标签扩展，而非确认流。）

**步骤4：关联对象（仅扫墓）**

- 从已故成员列表中选择祭扫对象
- 可多选（一次祭扫多位先人）
- 选择后显示：姓名、生卒年、墓碑照片（如有）

**步骤5：上传素材**

- 支持照片和视频（视频V1.5支持）
- 最多9个媒体文件
- 自动按时间/人物分类
- 支持添加文字说明

**步骤6：记录心得与活动侧素材（参与者维度）**

- **参与范围**：凡已被列入本次活动「参与人员」的成员，均可针对**本次活动**发布内容。
- **内容与形态**：每位参与者可**发布心得（文字）**、**上传与本次活动相关的照片**；允许在活动时间前后**多次补充**（每次提交生成独立记录或版本，由实现选型，**默认不采用「同一字段后写覆盖先写」跨成员生效**）。
- **不互覆盖原则**：**不同成员**提交的心得与照片**相互独立保存**；**同一成员**多次提交默认**追加为多条记录**（或提供「编辑本人上一条」仅影响本人该条，**不得覆盖他人记录**）。
- **可见性（可选能力）**：可配置「全员可见」「仅参与者可见」「仅自己可见」等策略；默认策略由产品另行拍板（可与动态墙可见性体系对齐）。
- **引导与字数**：可保留引导问题，如「这次活动有什么难忘的瞬间？」「有什么想对家人说的？」；单条心得字数上限建议 **200 字/条**（或与动态模块对齐），照片张数与压缩策略与活动相册章节一致。

> **与旧版 PRD 的差异：** 已删除「共同撰写 vs 各自填写」二选一且易导致**互顶覆盖**的描述；改为**参与者各自独立贡献、默认不互覆盖**。

#### 3.2 扫墓专项功能

**碑文转录：**
- 支持上传墓碑照片（最多3张）
- 提供文字输入框用于转录碑文内容
- 可设置碑文可见性（默认仅亲属可见）

**忌日提醒：**

| 提醒类型 | 设置项 |
|----------|--------|
| 周年忌日 | 日期（支持农历转换公历） |
| 清明节 | 自动计算当年清明日期 |
| 自定义日期 | 用户指定日期+重复周期 |

**提醒规则：**
- 提前3天推送首次提醒
- 当天早8点推送二次提醒
- 提醒内容：「[张三父亲的忌日] 明天是[张三父亲]的忌日，您有什么想记录或分享的吗？」

**祭品记录：**
- 预设祭品种类：香烛、纸钱、鲜花、供品、其他
- 支持自定义祭品名称
- 可上传祭品照片

**天气与氛围：**
- 自动获取活动当天当地的天气数据（温度、天气状况）
- 提供氛围标签选择：庄重、温情、热闹、感慨
- 记录参与人数和年龄段分布（可选）

#### 3.3 活动展示与回顾

**家族动态墙：**
- 融合个人动态与家族活动的综合信息流
- 活动卡片高亮显示（带活动图标）
- 支持按类型筛选（全部/仅活动/仅动态）

**活动相册：**
- 按活动分类的照片墙
- 时间轴展示（最近活动在上）
- 支持快速滑动浏览
- 可按参与成员筛选照片

**「那年今日」活动推送：**
- 推送过往同日期的家族活动回忆
- 格式：「[张三]，3年前的今天，你们进行了[清明扫墓]，[查看照片]」

## 模块四：记忆浏览与回顾

### 1. 功能概述与价值

记忆浏览模块提供多维度的内容检索和回顾功能，帮助用户在海量记录中快速找到想看的内容，通过年度回顾和「那年今日」等功能唤起珍贵记忆，增强家族情感连接。

### 2. 用户故事（User Story）

| # | 角色 | 故事描述 | 验收条件 |
|---|------|----------|----------|
| US-M01 | 怀旧的用户 | 作为想回顾往事的用户，我希望按成员查看完整的个人时间轴，以便系统地了解某位家人的成长历程 | 点击家族成员，查看该成员所有动态记录 |
| US-M02 | 好奇的后辈 | 作为想了解家族历史的后辈，我希望按时间浏览家族大事件，以便了解家族发展脉络 | 选择时间段，查看该时间范围内的家族事件 |
| US-M03 | 记录发烧友 | 作为热衷记录的用户，我希望搜索关键词找到过往内容，以便快速定位特定记忆 | 输入关键词，验证搜索结果的准确性和完整性 |
| US-M04 | 家族管理者 | 作为家族管理者，我希望查看年度回顾报告，以便向家族成员展示一年的共同回忆 | 每年末生成《我们的2026》年报，支持分享 |

### 3. 功能流程与交互细节

#### 3.1 多维度浏览

**按成员浏览：**
- 入口：家族树 → 点击成员 → 「查看时间轴」
- 展示：该成员所有动态记录（公开+亲属可见级别）
- 筛选：可按时间范围、标签类型进一步筛选

**按时间线浏览：**
- 入口：动态墙 → 切换为「时间线」模式
- 展示：家族大事件（融合个人动态与家族活动）
- 粒度：可选择「全部」「今年」「近三月」「自定义时间段」

**按活动类型筛选：**
- 入口：活动Tab → 筛选器
- 类型：扫墓、家庭聚会、节庆、婚礼寿宴、旅行、自定义
- 展示：该类型下所有活动及关联照片

#### 3.2 搜索功能

**搜索范围：**
- 动态内容（文字摘要）
- 成员姓名
- 活动时间
- 活动标题

**搜索规则：**
- 关键词匹配（支持模糊搜索）
- 按相关度排序
- 高亮匹配关键词
- 搜索历史记录（最近10条，可清除）
- 空结果友好提示：「未找到相关记录，试试其他关键词？」

#### 3.3 年度回顾（V1.5功能，V1.0仅做基础框架）

**年报内容框架：**
- 封面：《我们的2026》家庭年报
- 家族概况：成员总数、新增成员、活跃成员数
- 年度高频词：基于动态内容提取TOP10热词
- 重要时刻：按月份展示年度重大事件
- 足迹地图：家族成员本年度的活动地点分布
- 记录统计：总动态数、总照片数、人均记录数
- 寄语征集：邀请每位成员写下新年寄语

**年报生成与分享：**
- 自动生成时间：每年1月1日（基于上一年数据）
- 生成后推送通知：「您的《我们的2025》年报已生成」
- 展示形式：长图文，支持左右滑动翻页
- 分享：可分享至微信对话，生成海报图

### 4. 边界情况与错误处理

| 场景 | 处理方式 |
|------|----------|
| 搜索结果为空 | 展示空状态页，提供搜索建议（热门成员/最近动态入口） |

### 5. 验收标准（Acceptance Criteria）

## 模块五：数据与隐私

### 1. 功能概述与价值

数据与隐私模块保障用户数据安全，支持数据导出和账号继承，确保家族记忆的长期保存和可延续性。

### 2. 用户故事（User Story）

| # | 角色 | 故事描述 | 验收条件 |
|---|------|----------|----------|
| US-P01 | 数据安全意识强的用户 | 作为关注数据安全的用户，我希望所有数据云端加密备份，以便在更换手机后不丢失数据 | 验证数据云端同步和加密存储 |
| US-P02 | 家族创建者 | 作为创建者，我希望导出PDF家庭年鉴，以便离线保存家族记忆 | 导出PDF格式的家庭年鉴，包含成员信息和关键动态 |
| US-P03 | 担心账号失效的用户 | 作为担心年迈父母账号失效的用户，我希望指定守护人接管数据，以便家族记忆延续 | 设置守护人，验证权限转移流程 |

### 3. 功能流程与交互细节

#### 3.1 数据备份与导出

**自动云端备份：**
- 所有数据实时同步至微信云开发后端
- 传输全程HTTPS加密
- 本地缓存：关键数据（如家族树结构）缓存至本地，优先读取本地，辅以云端同步

**PDF导出：**
- 导出内容：家族成员列表（姓名/照片/关系/生卒年）+ 年度重要事件 + 家族合照
- 导出格式：A4竖版，封面+目录+内容页
- 导出范围：可选择导出全部内容或仅导出指定成员/时间范围
- 文件大小：预估10-50MB，压缩后提供下载链接

#### 3.2 账号与继承

**守护人设置：**
- 入口：我的 → 家族设置 → 守护人管理
- 可设置1-2位守护人（需对方同意）
- 守护人权限：管理所有成员数据、审核加入申请、修改家族设置
- 守护人权限不含：解散家族、转让创建者

**继承触发条件：**
- 创建者连续90天未登录
- 创建者主动申请账号注销
- 创建者账号被判定为失效（微信侧通知）

**继承流程：**
1. 系统向守护人发送通知：「创建者账号可能已失效，是否接管家族管理？」
2. 守护人确认接管 → 获得创建者权限
3. 原创建者账号登录后提示：「您的守护人已接管管理权，是否恢复？」

### 4. 边界情况与错误处理

| 场景 | 处理方式 |
|------|----------|
| PDF导出数据量过大 | 分卷导出（按成员分册），单个PDF≤50MB |
| 守护人拒绝接管 | 通知次级守护人（如有），无守护人则向所有管理员发送通知 |
| 创建者注销账号后恢复登录 | 提示「您的管理权已被守护人接管」，需守护人主动让回或管理员投票决定 |
| 数据导出时网络中断 | 本地暂存导出任务，网络恢复后自动继续 |
| 守护人也是普通成员 | 守护人身份优先级高于普通成员，可同时保留成员身份 |

### 5. 验收标准（Acceptance Criteria）

| AC编号 | 验收条件 | 测试方法 |
|--------|----------|----------|
| AC-P01 | 所有数据云端加密备份，换设备后数据完整同步 | 更换设备登录，验证家族树、动态等数据完整性 |
| AC-P02 | PDF导出正常生成，内容完整（成员+动态+照片） | 导出PDF，验证内容与实际数据一致 |
| AC-P03 | 守护人设置≤2人，支持更换和取消 | 设置守护人、变更守护人、取消守护人，验证权限变化 |
| AC-P04 | 继承触发后守护人正常接管，权限完整 | 模拟继承场景，验证守护人接管后的权限范围 |

### 4.2.2 亲属可见性计算规则

判断用户B是否可查看用户A的「亲属可见」内容：
1. 检查A与B是否为直系亲属（父母/子女/配偶）→ 是则通过
2. 检查A与B是否为旁系亲属（通过血缘关系链路≤3层）→ 是则通过
3. 检查B是否为A家族的管理员/守护人 → 是则通过
4. 否则，拒绝访问

# 附录A 竞品分析简报

## A.1 竞品概览

| 产品 | 平台 | 核心功能 | 差异化 |
|------|------|----------|--------|
| 族谱星球 | APP | 族谱管理、寻根问祖 | 侧重历史追溯，静态记录为主 |
| 亲族 | 微信小程序 | 家族动态、任务管理 | 侧重任务协作，社交属性强 |
| 族谱家 | APP | 族谱编辑、打印服务 | 侧重传统族谱数字化 |
| 家族相册 | 微信小程序 | 照片共享、相册管理 | 侧重照片管理，无动态记录 |
| 族脉记（我们） | 微信小程序 | 动态记录+族谱+活动 | 聚焦「活态传承」，情感连接 |

## A.2 竞争优势

1. **动态记录创新：** 区别于静态族谱，强调持续记录和情感连接
2. **扫墓专项功能：** 现有产品普遍缺失的文化场景支持
3. **关系类型扩展：** 支持结婚/结拜/认亲，贴近现代家庭实际
4. **轻量化：** 微信小程序，无需下载，降低使用门槛
5. **情感化设计：** 周期提醒、「那年今日」等情感化功能

## A.3 竞争风险

1. 用户习惯培养成本高（需要持续记录）
2. 家族成员集体迁移难度大（需要多人同时使用）
3. 微信生态内竞争（亲族等同类产品）

# 附录C 风险评估与应对策略

## C.1 风险矩阵

| 风险点 | 可能性 | 影响 | 风险等级 | 应对策略 |
|--------|--------|------|----------|----------|
| 用户参与度低，记录中断 | 高 | 高 | 🔴 严重 | 设计极简记录流程；强化智能提醒与情感化推送 |
| 隐私安全担忧 | 中 | 高 | 🟠 高 | 清晰的隐私协议；透明的权限控制说明；提供「仅自己可见」选项 |
| 功能复杂导致长辈不会用 | 中 | 中 | 🟡 中 | 推出「长辈简易模式」，隐藏高级设置，核心功能一步直达 |
| 同质化竞争 | 中 | 中 | 🟡 中 | 坚持「动态记录」与「活动场景」的差异化定位，快速迭代 |
| 技术实现难度超预期 | 中 | 高 | 🟠 高 | MVP阶段聚焦核心功能，技术预研提前启动 |
| 微信政策变化影响功能 | 低 | 高 | 🟠 高 | 持续关注微信政策，保持合规；核心功能有降级方案 |
| 服务器成本超出预期 | 低 | 中 | 🟢 低 | 初期使用微信云开发，按需扩容；V1.5评估自建后端 |

## C.2 启动策略

1. **种子用户获取：** 从个人网络邀请，建立首批完整家族树（≥3个家族）
2. **新手激活：** 通过「新手任务」引导用户完成创建家族、添加3位成员、完成首次记录
3. **留存策略：** 依赖智能周期提醒和「回忆推送」唤醒用户
4. **口碑传播：** 鼓励用户分享「那年今日」至微信对话，触达潜在用户
